# AI-Native 前端框架

让你自己的前端应用，获得高准确率、可见可控的 AI 操作能力。

> 完整定位、差异化、里程碑见 `docs/rfcs/0001-ai-native-frontend-framework.md`。开工前先读它。

## 这是什么

一个开源、长期维护的前端框架。核心：**构建时用 AST 从代码已有结构（路由、表单、组件）提取「能力清单」**，让 LLM 在编译期确定的能力白名单里精确编排操作，并在页面上可见地执行（虚拟光标、逐字填表、可中断）。

不做任意第三方网站自动化——只服务开发者掌控源码的应用。

## 当前阶段

阶段 0（RFC）已完成。下一步进入阶段 1：从 `ai-native-demo` 抽取可 npm 安装的 React 库。详见 `ROADMAP.md`。

## 目录约定

计划采用 monorepo（阶段 1 落地，届时定具体工具）：

```
docs/rfcs/        RFC 与设计决策记录
packages/         各发布包（core / scanner / preset-* / react / vue / cli）
examples/         示例应用（可能引用 ai-native-demo）
```

当前仅有 `docs/`，`packages/` 等在阶段 1 建立。

## 核心原则（框架特有）

1. **包边界清晰**：每个包能不看内部实现就说清「做什么、怎么用、依赖什么」。`core` 不认识任何前端框架，只认识「能力清单 + adapter 接口」。
2. **接入光谱**：自动推断（零改动）→ 配置补漏（`ai.config.ts`）→ 手动精标（`data-ai-*`）。低地板 + 高天花板，不二选一。
3. **诚实的边界**：自动推断准确率 < 手动标注，是物理规律。不宣称「全自动且最准」。
4. **开源纪律**：公共 API 变更走语义化版本；破坏性变更进 CHANGELOG；面向社区的决策留记录。

## 关键约束

- **对齐 RFC**：每次重大技术决策，回看 RFC 的定位与非目标（§6），确认不跑偏；变更了定位就先改 RFC。
- **测试先行**：框架代码质量门槛高于应用，实现前先写测试（TDD）。
- **拥抱 MCP，不重造传输协议**；不引入纯视觉识别。
- **公开发布是红线**：npm publish、正式 release 前必须停下确认。

## 与 ai-native-demo 的关系

`../ai-native-demo` 是本框架的**原型来源**和**首个使用者**。框架的 `core`/`scanner` 从 demo 的 `src/ai`、`scripts/ai-scanner` 抽取泛化。阶段 1 会用 demo 反过来验证框架 API 是否好用。两者是独立仓库。
