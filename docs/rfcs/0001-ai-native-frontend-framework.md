# RFC 0001：AI-Native 前端框架

- **状态**：草案（Draft）
- **日期**：2026-07-16
- **作者**：FantasticPerson
- **目标读者**：框架潜在使用者、贡献者、以及未来的自己

> 本文档是定位与差异化 RFC，不是实现规格。目的：在投入数月开发前，先把「凭什么做、和谁不一样、边界在哪、分几步走」想清楚。如果这份文档立不住，就说明现在还不该做这个框架。

---

## 1. 定位与一句话主张

**让你自己的前端应用，获得高准确率、可见可控的 AI 操作能力。**

用户对着应用说一句话（「帮我提个明天的事假」「把报销按差旅筛选」），AI 自动完成对应的多步操作——切页、填表、点击、提交——并且全程**看得见**：虚拟光标移动、逐字填写、目标高亮，用户能看懂、能信任、能随时中断。

关键限定词，决定了整个框架的边界：

- **「你自己的应用」**——只服务开发者能掌控源码的应用（中后台、SaaS、管理系统），不做任意第三方网站的自动化。
- **「高准确率」**——这是核心护城河，来自编译期就确定的能力边界，而非运行时的视觉猜测。
- **「可见可控」**——执行过程是透明的产品体验，不是黑盒。

一句话区分竞品：**别人让 AI「看屏幕猜着操作任意网站」，我们让 AI「读结构精确操作你自己的应用」。**

---

## 2. 核心洞察

**现代前端应用的语义早已存在，只是散落各处、从未被收集。**

一个典型的 React 中后台项目里：

- **有哪些页面**——写在 react-router 的路由配置里。
- **每个表单有哪些字段、什么类型、是否必填、校验规则**——写在 antd `<Form.Item>`、react-hook-form 的 `register`、或 zod/yup schema 里。
- **有哪些可点击的操作**——写在 `<button>`、`<a>`、带 `onClick` 的组件上，通常还带着可读的文案。

这些信息本就是「应用能做什么」的权威定义。问题不在于「缺少语义」，而在于**没有一个东西把它们收集起来，变成 AI 能读懂的能力清单**。

因此本框架的核心动作是：**在构建时用 AST 静态分析，从代码已有的结构中提取「能力清单」（manifest）**，而不是要求开发者为 AI 重新标注一遍。标注是可选的增强手段，不是接入的前提。

> 这一点是从当前 demo 反推出来的升级。demo 的做法是「自己发明 `data-ai-*` 属性，让开发者手贴」；本框架把同一套 AST 扫描技术，转向**扫描已经存在的路由与表单结构**——接入成本因此从「逐个标注」降到「多数场景零改动」。

## 3. 差异化对比

要开源，就必须回答「凭什么用你，而不是已有方案」。三类竞品：

### 3.1 vs 浏览器 Agent（browser-use / OpenAI Operator / Anthropic Computer Use）

这类方案靠**视觉截图或 DOM 快照**，让模型推断「哪个元素能点」。

| 维度 | 浏览器 Agent | 本框架 |
|---|---|---|
| 元素定位 | 运行时视觉/DOM 推断，受渲染、遮挡、动态布局影响 | 编译期确定的能力白名单，LLM 只在封闭集合里选 ID |
| 准确率 | 会点错、点到不存在的元素 | 白名单校验，越界即拒（见 `parsePlan`） |
| Token 成本 | 需传视觉/大段 DOM | 只传结构化清单，省 token |
| 可审计 | 能力边界不确定 | 能力边界编译期已知，可审计、可白名单 |
| 适用范围 | **任意网站**（含无源码的第三方） | **仅自有应用**（能被扫描/标注的） |

**这不是竞争，是互补。** 它们解决「自动化我控制不了的网站」，我们解决「让我自己的产品拥有最准的 AI 操作能力」。准确率是我们的主场，通用性是它们的主场。

### 3.2 vs MCP（Model Context Protocol）

MCP 是 agent 连接外部能力的传输标准，正在成为事实标准。**它和本框架不在同一层，不冲突，甚至应该结合**：

- MCP 解决「agent 如何调用一个能力」的传输与协议问题。
- 本框架解决「前端应用有哪些能力、如何从代码提取、如何在页面上可见地执行」的问题。

未来方向：本框架产出的能力清单，可以**作为一个 MCP server 暴露**，让任意支持 MCP 的 agent 都能操作你的应用。不重造传输层。

### 3.3 vs ARIA / 无障碍树

ARIA 是通用可访问性语义（role、label、状态），运行时就存在。差异：

- ARIA 描述「这个元素是什么」（给读屏软件用），**不描述「这是一个可被 AI 编排的业务操作」**。
- 本框架的清单是**面向可执行操作**的：一个操作需要先导航到哪个模块、填哪些字段、字段有什么约束、点哪个按钮提交——这是编排信息，ARIA 没有。

实践上，ARIA 是**运行时兜底提取的信号源之一**（见 §4 第三层），但不是全部。

## 4. 三层接入光谱

框架的关键设计：**接入成本与准确率是一条光谱的两端，不是二选一。** 开源框架若没有「零改动就能试」的低地板，再高的准确率天花板也没人愿意爬。

```
接入成本地板  ←──────────────────────────────────→  准确率天花板
  零改动                                              精细标注

  ① 自动推断            →      ② 配置补漏      →      ③ 手动精标
  preset 扫路由/表单/DOM       ai.config.ts           data-ai-* 标注
  覆盖大部分场景               在一处修正漏判/歧义     歧义清零、最高准确率
```

### 第一层：自动推断（零改动，覆盖大部分场景）

通过 preset（预设适配器）从现有代码提取，按信息质量从高到低：

1. **扫路由配置** → 得到「模块清单」。react-router routes、vue-router、文件路由。
2. **扫组件库表单** → 得到「字段清单」。antd `<Form.Item name label rules>`、react-hook-form `register`、zod/yup schema——字段名、label、类型、必填、校验规则往往比手贴的标注还全。
3. **运行时扫 DOM + ARIA** → 兜底覆盖。`<button>`、`<input>`+`<label>`、`aria-label`、`placeholder`、`<a href>`。零改动，但只能看到已渲染的元素。

### 第二层：配置补漏（改动集中在一个文件）

自动推断必然有歧义（两个都叫「提交」的按钮、动态生成的表单）。用**一个 `ai.config.ts`** 在集中一处修正、消歧、补充，不必回到每个业务组件里改。

### 第三层：手动精标（天花板）

对核心高频路径，用 `data-ai-*` 显式标注（即当前 demo 的机制），获得最高准确率，歧义清零。

**诚实的边界**：自动推断准确率 < 手动标注，这是物理规律。框架不宣称「全自动且最准」，而是提供一条从「零改动跑起来」到「精标到极致」的平滑路径，让用户按需投入。

## 5. 架构分层

沿用并推广当前 demo 已验证的三层解耦（业务 / 提取 / 运行，靠能力清单契约串联），拆成可独立发布的包：

```
@ai-operable/core         框架无关的运行时：解析 LLM 输出、白名单校验、执行器、虚拟光标
                        （从 demo 的 src/ai 抽取，剥离 React/react-router 依赖）

@ai-operable/scanner      构建时 AST 提取引擎 + manifest 生成 + 构建插件
                        （从 demo 的 scripts/ai-scanner 抽取、泛化）

@ai-operable/preset-*      各栈的提取预设：preset-react-router、preset-antd …
                        （新建，是「自动推断」的载体，也是长期维护重点）

@ai-operable/react        React 适配器：AIBar 组件、useAIAgent hook、与 router 的桥接
@ai-operable/vue          Vue 适配器（阶段 3 才做，用于反证内核真的解耦）

@ai-operable/cli          脚手架、扫描调试、manifest 预览
```

**来源标注**：

- **可直接抽取**：`parsePlan` 白名单校验逻辑、executor 的 DOM 操作与虚拟光标、AST 扫描骨架——这些 demo 里已验证、已有单测。
- **需重构**：executor 现在绑死 react-router 的 `navigate` 和 React 受控组件的 setter，要抽象出 adapter 接口才能进 `core`。
- **全新建**：preset 体系、`ai.config.ts` 配置层、LLM provider 抽象（现在写死 DeepSeek）、执行结果回流 LLM 的反馈闭环、CLI。

各包边界的检验标准：**能不看内部实现就说清它做什么、怎么用、依赖什么。** 比如 `core` 不认识任何前端框架，只认识「能力清单 + adapter 接口」；换掉 React 适配器不影响 `core`。

## 6. 目标 / 非目标

### 目标

- 服务**自有应用**的 AI-native 化（中后台、SaaS、管理系统优先）。
- **React 生态优先**，首个验证栈：React + react-router + antd。
- 提供**三层接入光谱**，低地板 + 高天花板。
- **可见可控执行**作为一等体验：虚拟光标、逐步演出、可中断。
- 内核**框架无关**，为多框架适配预留接口。

### 非目标（明确不做，避免范围蔓延）

- **不做任意第三方网站的自动化**——那是浏览器 Agent 的主场。
- **不做纯视觉 fallback**——运行时 DOM/ARIA 提取只是兜底信号，不引入视觉识别，否则丢掉结构化优势且打不过专门做视觉的对手。
- **1.0 不做多框架**——先把 React 一条链路做到最好，Vue adapter 仅用于阶段 3 反证解耦。
- **不重造传输协议**——拥抱 MCP，不发明新的 agent 通信标准。
- **暂不做商业化 / 云服务设计**——现在写这些都是猜测，YAGNI。

## 7. 里程碑

每个阶段有可验证的产出，不一步到位奔 1.0。

| 阶段 | 目标 | 可验证产出 | 风险 |
|---|---|---|---|
| **0 · RFC** | 定位与差异化想清楚 | 本文档；能说服自己和早期读者「凭什么做」 | 立论不成立就停 |
| **1 · React 库自用** | 从 demo 抽出可 npm 安装的 React 库 | `core`+`scanner`+`react`+`preset-react-router`；**用当前 demo 作为首个使用者**验证 API 手动标注链路跑通 | API 设计不合理 |
| **2 · 内核去框架化 + 自动推断** | 剥离 React 依赖，落地第一层 preset | `core` 不含 React；`preset-antd` 能零改动扫出字段；LLM provider 可插拔；执行反馈闭环 | 解耦不彻底 |
| **3 · Vue adapter 反证** | 用第二个框架证明内核真解耦 | `@ai-operable/vue` 跑通同一套 core；两个 adapter 并存 | 内核仍偷偷依赖 React |
| **4 · 文档站 + 治理 + 发布** | 具备开源社区门槛 | 文档站、可交互 playground、LICENSE、CONTRIBUTING、语义化版本、CI | 维护投入持续性 |

**阶段间的关卡**：每阶段结束前，回看本 RFC 的定位与非目标，确认没有跑偏。**阶段 4 的公开发布是不可逆操作，届时会停下来单独确认。**

## 附录 A：待解决的开放问题

以下问题现在无需答案，但需在对应阶段前解决，先记录避免遗忘：

1. **preset 的维护成本**：每个框架/组件库一个适配器，还要跟版本升级。优先级如何排？社区如何共建？（阶段 2 前）
2. **执行反馈闭环的设计**：执行失败后如何把结果回流给 LLM 重规划？多轮对话如何管理状态？（阶段 2）
3. **动态表单/条件渲染**：字段随交互出现，静态扫描看不到，如何与运行时提取协同？（阶段 2）
4. **安全模型**：AI 能执行的操作是否需要权限分级？危险操作（删除、提交）是否需要二次确认？（阶段 1 就要有基本设计）
5. **命名与品牌**：~~`@ai-native/*` 是占位，正式名称待定。~~ **已定（阶段 4）**：正式 scope 定为 `@ai-native`。理由——npm 上整个 `@ai-native` scope 完全可用（core/scanner/react/vue/preset-* 全部未被占用），语义精准，且已贯穿 6 包 + 2 demo + 全部文档，转正零返工。另起新名要批量改却无收益。**更正（阶段 4 发布时）**：真正发布时发现 `@ai-native` 的 npm org 已被他人注册、当前账号无发布权（此前只验证过包名未被占用，未验证 scope 的发布权限），故正式 scope 改为 **`@ai-operable`**（对应已创建的 npm org），6 包 + 2 demo + 全部文档随之更新。教训：scope 可用性 = 名字未占用 **且** 有发布权，二者都要在发布前验证。
