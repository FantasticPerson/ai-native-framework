# 为什么做

## 一句话主张

**让你自己的前端应用，获得高准确率、可见可控的 AI 操作能力。**

用户对着应用说一句话（「帮我提个明天的事假」「把报销按差旅筛选」），AI 自动完成对应的多步操作——切页、填表、点击、提交——并且全程看得见。

## 核心洞察

现代前端应用的语义早已存在，只是散落各处、从未被收集：

- **有哪些页面**——写在路由配置里。
- **每个表单有哪些字段、什么类型、是否必填**——写在 `<Form.Item>`、`register`、zod/yup schema 里。
- **有哪些可点击的操作**——写在带文案的 `<button>` / `<a>` 上。

这些信息本就是「应用能做什么」的权威定义。问题不在于缺少语义，而在于没有一个东西把它们收集起来，变成 AI 能读懂的能力清单。

因此本框架的核心动作是：**在构建时用 AST 静态分析，从代码已有的结构中提取「能力清单」（manifest）**，而不是要求开发者为 AI 重新标注一遍。

## 和谁不一样

一句话区分竞品：**别人让 AI「看屏幕猜着操作任意网站」，我们让 AI「读结构精确操作你自己的应用」。**

- **vs 浏览器 Agent（browser-use / Operator / Computer Use）**：它们靠视觉猜测操作任意网站，通用但准确率受限；我们只服务你掌控源码的应用，用编译期确定的能力边界换高准确率。
- **关键限定词**：只服务开发者能掌控源码的应用（中后台、SaaS、管理系统），不做任意第三方网站自动化。

## 诚实的边界

- 自动推断准确率 < 手动标注，是物理规律。我们不宣称「全自动且最准」。
- 拥抱 MCP，不重造传输协议；不引入纯视觉识别。

> 完整定位与差异化对比见 [RFC 0001](https://github.com/FantasticPerson/ai-native-framework/blob/main/docs/rfcs/0001-ai-native-frontend-framework.md)。
